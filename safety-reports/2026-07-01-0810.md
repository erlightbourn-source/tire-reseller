## (a) Fixed — committed on `de/safety-20260701`

### `13494fc` — Apply the soft-deleted-seller visibility filter to all public listing feeds
- **Severity:** medium (data-integrity / defense-in-depth)
- **Files:**
  - `app/browse/page.js` — results, cohort avg, brand-filter list, "similar sizes" count
  - `app/page.js` — homepage feeds + stats (count, groupBy, recent, brand rows, showcase fallback)
  - `app/sizes/[size]/page.js` — size landing listings
  - `app/tires/[brand]/page.js` — brand resolver + brand listings
  - `app/listings/[id]/page.js:87` — "More like this" candidates
  - `app/sell-tires/page.js:29` — active-listing counter
  - `app/sitemap.js` — brand / size / listing URL emitters
- **Impact:** Notifications (`app/api/notifications/route.js:42`) and the alerts cron (`app/api/cron/alerts/route.js:36,79`) already enforce the public-visibility contract "show only `hidden: false` **AND** `seller.deletedAt = null`" — a deleted/banned seller's listing 404s when opened, so the listing-detail page already 404s it. Browse, homepage, size/brand landing pages, the sell-tires counter, and the sitemap were only filtering `hidden: false`, relying on soft-delete *also* flipping listings to hidden. That coupling is fragile: `admin/route.js:35` and `login/route.js:55` both clear `deletedAt` (unban / reactivation) **without** re-hiding, and the sitemap was emitting listing URLs that could 404 for search engines. The fix adds `seller: { deletedAt: null }` at each public call site. The unit-tested `buildListingWhere` helper was left untouched. Behavior-preserving for legitimate (non-deleted) sellers.
- **Verification (before commit, re-confirmed independently this run):**
  - `npm test` → **61/61 pass**, 0 fail
  - `APP_SECRET=… npm run build` → succeeded, full route table compiled, no errors
  - `prisma/schema.prisma` line 9 restored to `provider = "postgresql"` (not in diff)
  - Diff is uniform: every hunk only adds `seller: { deletedAt: null }` to an existing `where`

`npm audit` (prod): **0 vulnerabilities**.
